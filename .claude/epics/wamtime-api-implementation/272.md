# Task: Function Invocation Implementation

## Objective
Complete the WebAssembly function calling mechanism in native Rust code, enabling Java applications to invoke WebAssembly functions with proper parameter marshalling and return value handling. This builds on the Store context integration to make function calls actually work.

## Implementation Details

### Function Call Pipeline
- Implement complete function invocation in `wasmtime4j-native/src/jni_bindings.rs`
- Add support for all WebAssembly parameter types (i32, i64, f32, f64, v128, funcref, externref)
- Implement return value marshalling from WebAssembly to Java types
- Add proper error propagation from WebAssembly traps to Java exceptions

### Parameter Marshalling
- Convert Java primitive types to WebAssembly value types safely
- Handle Java arrays for bulk parameter passing
- Implement reference type handling for funcref and externref parameters
- Add validation for parameter count and type compatibility
- Support parameter conversion with proper bounds checking

### Return Value Handling
- Marshal WebAssembly return values to appropriate Java types
- Handle multiple return values (WebAssembly 1.0 multi-value proposal)
- Implement proper error handling for type conversion failures
- Add support for void functions (no return values)
- Ensure memory safety during return value conversion

### Error Propagation
- Map WebAssembly traps to appropriate Java exceptions
- Implement stack trace preservation from WebAssembly to Java
- Add meaningful error messages for function call failures
- Handle timeout and resource exhaustion scenarios during function calls

### Performance Optimization
- Minimize JNI overhead for function parameter marshalling
- Implement efficient conversion for array parameters
- Cache function signatures for repeated calls
- Optimize common parameter type patterns

## Acceptance Criteria
- [ ] Simple WebAssembly functions (add, multiply) can be called successfully
- [ ] All WebAssembly parameter types are supported (i32, i64, f32, f64)
- [ ] Functions with multiple parameters work correctly
- [ ] Functions with multiple return values are handled properly
- [ ] WebAssembly traps are converted to appropriate Java exceptions
- [ ] Function calls with invalid parameters fail gracefully with clear errors
- [ ] Performance overhead is minimal (<100μs for simple function calls)
- [ ] Memory safety is maintained during all parameter/return value conversions

## Technical Notes

### Wasmtime Function API
- Use Wasmtime 36.0.2 typed function APIs for type safety
- Implement proper function instance management
- Follow Wasmtime calling conventions for parameter passing
- Handle Wasmtime errors according to API documentation

### JNI Function Integration
- Implement efficient JNI array access for bulk parameters
- Use appropriate JNI types for WebAssembly value representation
- Handle JNI exceptions during parameter/return value conversion
- Minimize JNI call overhead through batching where possible

### Type System Integration
- Map WebAssembly types to Java types consistently
- Handle type widening/narrowing conversions safely
- Implement proper validation for reference types
- Add support for custom type conversion hooks

### Memory Management
- Ensure parameter arrays are properly released after function calls
- Handle temporary native allocations during type conversion
- Prevent memory leaks in error scenarios
- Implement proper cleanup for partially constructed parameters
