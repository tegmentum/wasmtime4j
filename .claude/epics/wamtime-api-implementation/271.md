# Task: Store Context Integration

## Objective
Fix the fundamental Store context lifecycle and threading issues in native Rust code to enable all WebAssembly operations. This task addresses the core architectural problem preventing function calls, memory access, and WASI operations from working correctly.

## Implementation Details

### Store Lifecycle Management
- Fix Store creation and destruction in `wasmtime4j-native/src/store.rs`
- Implement proper thread-safe context isolation between WebAssembly instances
- Add Store-scoped resource tracking for automatic cleanup of native resources
- Ensure Store contexts are properly linked to Java objects through JNI handles

### Threading Safety
- Implement thread-local storage for Store contexts where required by Wasmtime
- Add synchronization mechanisms for multi-threaded access patterns
- Ensure Store contexts cannot be accessed across incompatible thread boundaries
- Implement proper error handling for thread-related Store operations

### Resource Tracking
- Add Store-scoped tracking for all WebAssembly resources (functions, memory, modules)
- Implement automatic cleanup when Store is disposed or garbage collected
- Add reference counting for shared resources across multiple Store instances
- Ensure native memory leaks are prevented through proper lifecycle management

### JNI Integration
- Fix JNI handle management for Store objects in `jni_bindings.rs`
- Implement proper Java-to-native Store context mapping
- Add validation for Store context validity before all operations
- Ensure Store disposal from Java properly cleans up native resources

## Acceptance Criteria
- [ ] Store objects can be created successfully without native crashes
- [ ] Store contexts maintain isolation between different WebAssembly instances
- [ ] Multiple Store instances can exist simultaneously without interference
- [ ] Store disposal from Java properly releases all native resources
- [ ] Store operations are thread-safe according to Wasmtime's threading model
- [ ] No memory leaks detected after Store creation/destruction cycles
- [ ] Store context validation prevents operations on invalid/disposed stores

### Integration Tests

## Technical Notes

### Wasmtime Store API
- Study Wasmtime 36.0.2 Store documentation for proper usage patterns
- Understand Store thread affinity requirements
- Follow Wasmtime best practices for Store lifecycle management
- Implement error handling according to Wasmtime API specifications

### JNI Considerations
- Use weak global references for Store objects to prevent GC issues
- Implement proper exception handling for Store operations
- Ensure JNI calls are safe across different thread contexts

### Performance Considerations

- Cache Store context lookups where safe to do so
