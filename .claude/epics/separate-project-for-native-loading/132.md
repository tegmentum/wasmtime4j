# Task 003: Parameterize NativeLibraryUtils

## Objective
Extract and parameterize the 3 hardcoded strings in NativeLibraryUtils to enable configuration for different library projects while maintaining backward compatibility.

## Background
The current NativeLibraryUtils class has hardcoded string constants that make it specific to wasmtime4j. For the separate native loading project, these values need to be configurable while maintaining the existing API.

## Requirements

### Functional Requirements
- Extract hardcoded strings into configurable parameters
- Maintain backward compatibility with existing API
- Support configuration through constructor or builder pattern
- Preserve all existing functionality and behavior

### Technical Requirements
- Parameterize these 3 hardcoded strings:
  - `LIBRARY_NAME = "wasmtime4j"` (line 51)
  - `TEMP_FILE_PREFIX = "wasmtime4j-native-"` (line 54) 
  - `TEMP_DIR_SUFFIX = "-wasmtime4j"` (line 57)
- Add configuration class or builder to set these values
- Default values must match current hardcoded values
- All existing public methods must continue to work unchanged
- Thread-safe configuration handling

### Performance Requirements
- No performance degradation in library loading
- Minimal memory overhead for configuration
- Maintain existing caching behavior

### Security Requirements
- Validate all configurable parameters to prevent path traversal
- Sanitize library names and path components
- Maintain existing security protections

## Implementation Notes

### Configuration Approach
Consider these design options:
1. Constructor overloads with parameter objects
2. Builder pattern for fluent configuration
3. Static factory methods with configuration
4. System properties as fallback configuration

### Validation Strategy
- Sanitize library names using existing `sanitizeForLog` patterns
- Validate prefixes/suffixes contain only safe characters
- Prevent empty or null values where inappropriate
- Use existing security validation patterns

### Threading Considerations

- Consider using volatile or AtomicReference for dynamic config
- Maintain existing thread-safety guarantees
