# Task 005: Add Multi-Convention Resource Path Support

## Objective
Extend the native library loading system to support multiple resource path naming conventions (Maven Native, JNA, and custom patterns) to maximize compatibility with different build systems and library packaging approaches.

## Background
Different Java native library projects use various conventions for organizing native libraries in JAR resources. Supporting multiple conventions will make the separate native loading project more versatile and compatible with existing libraries without requiring changes to their build processes.

## Requirements

### Functional Requirements
- Support Maven Native Plugin convention: `/natives/{os}-{arch}/`
- Support JNA convention: `/{platform}/`  
- Support custom path patterns via configuration
- Maintain existing wasmtime4j path convention as default
- Try multiple conventions in priority order
- Log which convention succeeded for debugging

### Technical Requirements
- Extend NativeLoaderConfig to include path patterns
- Add PathConvention enum or similar abstraction
- Update resource path resolution logic
- Support fallback through multiple conventions
- Maintain backward compatibility with existing paths

### Path Convention Support
1. **Wasmtime4j Convention** (default): `/native/{platform}/`
2. **Maven Native Convention**: `/natives/{os}-{arch}/`  
3. **JNA Convention**: `/{platform}/`
4. **Custom Convention**: User-provided pattern with placeholders

### Performance Requirements
- Resource path resolution should be efficient
- Avoid excessive resource probing
- Cache successful convention per library
- Minimal overhead for default case

### Security Requirements
- Validate all path patterns to prevent traversal
- Sanitize platform/architecture strings in paths  
- Prevent malicious custom patterns
- Use existing security validation patterns

## Implementation Details

### Path Convention Abstraction
```java
public enum PathConvention {
    WASMTIME4J("/native/{platform}/"),
    MAVEN_NATIVE("/natives/{os}-{arch}/"),
    JNA("/{platform}/"),
    CUSTOM(""); // Pattern provided by user
}
```

### Configuration Extension
```java
public final class NativeLoaderBuilder {
    public NativeLoaderBuilder pathConvention(PathConvention convention);
    public NativeLoaderBuilder customPathPattern(String pattern);
    public NativeLoaderBuilder conventionPriority(PathConvention... order);
}
```

### Placeholder Support
- `{platform}` - full platform identifier (e.g., "linux-x86_64")
- `{os}` - operating system (e.g., "linux")  
- `{arch}` - architecture (e.g., "x86_64")
- `{library}` - library name
- Custom placeholders for specific use cases

### Resource Path Resolution Strategy
1. Get convention priority list from configuration
2. For each convention:
   - Generate resource path using convention pattern
   - Check if resource exists
   - If found, cache and return path
   - If not found, try next convention
3. If all conventions fail, provide diagnostic information

### Caching Strategy
- Cache successful convention per library name
- Use platform-specific cache keys
- Clear cache on configuration changes
- Optional cache warming for performance
