# Task #002: Implement Resource Lifecycle Management Infrastructure

## Objective

Implement comprehensive resource lifecycle management infrastructure for all JNI classes, establishing patterns for safe resource allocation, tracking, and cleanup to prevent memory leaks and ensure JVM stability.

## Background

The wasmtime4j-jni module manages complex native resources (engines, modules, instances, stores, functions) that require careful lifecycle management. This task creates the core infrastructure for tracking and managing these resources safely across all JNI implementations.

## Implementation Requirements

### 1. Resource Handle Management System
**File**: `wasmtime4j-native/src/jni/resource_handles.rs`
- Implement centralized handle allocation and tracking system
- Create type-safe handle wrappers for different resource types
- Add reference counting and ownership tracking
- Implement handle validation and lifecycle state management

### 2. Automatic Cleanup Infrastructure
**File**: `wasmtime4j-native/src/jni/cleanup_manager.rs`
- Create automatic resource cleanup mechanisms
- Implement finalizer integration for garbage collection scenarios
- Add orphaned resource detection and cleanup
- Create shutdown hook for application-level cleanup

### 3. Resource Tracking and Monitoring
**File**: `wasmtime4j-jni/src/main/java/ai/tegmentum/wasmtime4j/jni/ResourceTracker.java`
- Implement Java-side resource tracking for debugging
- Add metrics collection for resource usage patterns
- Create leak detection and reporting mechanisms
- Implement resource usage validation and limits

### 4. Cross-Reference Management
**File**: `wasmtime4j-native/src/jni/cross_references.rs`
- Implement safe cross-reference management between resources
- Handle dependency relationships (Instance->Store, Function->Store, etc.)
- Add circular reference detection and prevention
- Create dependency-aware cleanup ordering

## Technical Specifications

### Resource Handle System Design
```rust
pub struct ResourceHandle<T> {
    id: HandleId,
    resource: Option<T>,
    ref_count: AtomicUsize,
    state: AtomicResourceState,
    created_at: Instant,
}

pub enum ResourceState {
    Active,
    MarkedForCleanup,
    Cleaned,
    Invalid,
}

pub trait ManagedResource {
    fn cleanup(&mut self) -> Result<(), WasmtimeError>;
    fn is_valid(&self) -> bool;
    fn dependencies(&self) -> Vec<HandleId>;
}
```

### Cleanup Manager Infrastructure
```rust
pub struct CleanupManager {
    resources: Arc<RwLock<HashMap<HandleId, Box<dyn ManagedResource>>>>,
    cleanup_queue: Arc<Mutex<VecDeque<HandleId>>>,
    shutdown_hooks: Vec<Box<dyn FnOnce() + Send>>,
}

impl CleanupManager {
    pub fn register_resource<T: ManagedResource>(&self, resource: T) -> HandleId;
    pub fn schedule_cleanup(&self, handle_id: HandleId);
    pub fn force_cleanup_all(&self) -> Result<(), Vec<WasmtimeError>>;
    pub fn add_shutdown_hook<F>(&mut self, hook: F) where F: FnOnce() + Send + 'static;
}
```

## Implementation Notes

### Resource Safety Strategy
- Use RAII patterns extensively for automatic cleanup
- Implement double-cleanup protection (idempotent cleanup)
- Add extensive validation to prevent use-after-free scenarios
- Use atomic operations for thread-safe resource state management

### Memory Management Approach
- Minimize native memory allocation where possible
- Use reference counting for shared resources
- Implement weak references to break circular dependencies
- Add memory pressure notifications to trigger cleanup

### Error Handling Strategy
- Never fail silently during resource cleanup
- Log all cleanup failures for debugging
- Implement best-effort cleanup even when errors occur
- Provide detailed error information for troubleshooting

### Testing Strategy
- Test resource cleanup under normal operation
- Simulate garbage collection scenarios and finalizer calls
- Test cleanup during application shutdown
- Stress test with high resource allocation/deallocation rates
- Test circular dependency scenarios and cleanup ordering
