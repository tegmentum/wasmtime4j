# Task: Fix Runtime Discovery System

## Problem Statement

The wasmtime4j runtime discovery system is fundamentally broken, preventing any actual WebAssembly execution. The factory cannot locate JNI or Panama implementations despite them existing in the codebase.

**Critical Error:**
```
WARNING: JNI execution failed: Requested runtime type JNI is not available and no suitable fallback found
WARNING: PANAMA execution failed: Requested runtime type PANAMA is not available and no suitable fallback found
```

## Root Cause Analysis

1. **Missing Runtime Classes**: Factory cannot find:
   - `ai.tegmentum.wasmtime4j.jni.JniWasmRuntime`
   - `ai.tegmentum.wasmtime4j.panama.PanamaWasmRuntime`

2. **Module/Classpath Issues**: Implementation modules not properly discoverable

3. **Factory Registration Broken**: Runtime registration mechanism not working

## Technical Approach

### 1. Runtime Class Implementation
Create missing runtime entry point classes:
```java
// JniWasmRuntime.java
public class JniWasmRuntime implements WasmRuntime {
    @Override
    public String getRuntimeType() { return "JNI"; }

    @Override
    public Engine createEngine() {
        return new JniEngine();
    }
    // ... complete implementation
}
```

### 2. Factory Registration Fix
Implement proper runtime registration:
```java
public class WasmRuntimeFactory {
    static {
        // Auto-register available runtimes
        registerRuntime(new JniWasmRuntime());
        if (isPanamaAvailable()) {
            registerRuntime(new PanamaWasmRuntime());
        }
    }
}
```

### 3. Module Configuration
Fix module-path and classpath discovery:
- Ensure implementation modules are on classpath
- Add proper module declarations if needed
- Fix ServiceLoader registration if used

### 4. Native Library Loading
Ensure native libraries are properly loaded:
- Fix library path resolution
- Validate native symbol availability
- Handle loading failures gracefully
