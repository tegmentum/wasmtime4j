# Task: Complete Native-Java Bridge Integration

## Problem Statement

While native Rust libraries compile successfully and Java wrapper classes exist, the bridge between native code and Java implementations appears incomplete or disconnected. This creates a gap where individual components work in isolation but cannot communicate effectively.

**Key Issues:**
- Native libraries exist but Java classes can't access them properly
- JNI/Panama bindings may not be correctly connected
- Parameter marshaling and type conversion incomplete
- Error propagation from native to Java layer broken
- Resource management across native/Java boundary unclear

## Current State Analysis

### What Works ✅
- **Native Rust Implementation**: wasmtime4j-native compiles successfully
- **Java Interface Layer**: Well-designed APIs in wasmtime4j module
- **Wrapper Classes**: JNI and Panama wrapper classes exist and compile
- **Build System**: Cross-compilation produces native libraries

### What's Broken ❌
- **Runtime Discovery**: Factory can't find implementation classes
- **Native Symbol Access**: Java classes can't call native functions
- **Parameter Marshaling**: Complex type conversion incomplete
- **Error Handling**: Native errors not properly translated to Java exceptions
- **Memory Management**: Resource cleanup across boundaries unclear

```java
// Step 1: Fix Panama symbol lookup
public class PanamaSymbolResolver {

    private static final SymbolLookup WASMTIME_SYMBOLS =
        SymbolLookup.libraryLookup("wasmtime4j", SegmentScope.auto());

    public static MethodHandle resolveFunction(String name, FunctionDescriptor descriptor) {
        Optional<MemorySegment> symbol = WASMTIME_SYMBOLS.lookup(name);
        if (symbol.isEmpty()) {
            throw new RuntimeException("Native symbol not found: " + name);
        }
        return Linker.nativeLinker().downcallHandle(symbol.get(), descriptor);
    }
}
```
